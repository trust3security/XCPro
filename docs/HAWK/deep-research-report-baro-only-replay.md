# What “on baro‑only replay” Usually Means in Variometer and Flight‑Log Software

## Findings in brief

The phrase **“on baro‑only replay” is not a widely standardized, officially-defined feature name** across the major user manuals and public documentation for flight computers and variometer apps that support flight replay (e.g., XCSoar, LK8000, XCTrack). What *is* clearly standardized is **what “replay” and “baro” data mean in these ecosystems**, and those standards strongly constrain what “baro‑only replay” can plausibly refer to.

Based on how “replay” is implemented in mainstream glide/vario software and how altitude is represented in the **IGC** standard, “baro‑only replay” is best interpreted as shorthand for:

- **Replaying a flight log (typically an IGC file) while deriving altitude/vario from barometric/pressure altitude only** (i.e., ignoring GNSS altitude as the vertical reference), or  
- **Replaying a recorded data stream that contains only barometric altitude/vario** (e.g., “baro data only” device protocols), as opposed to replaying a full navigation solution.

It is **much less likely** to mean “replay raw barometer sensor samples (pressure in Pa)”, because **IGC logs don’t store raw pressure**; they store **pressure altitude** (and GNSS altitude) as derived values. citeturn6view1turn5view0

This matters for developer workflows: a “baro‑only replay” mode is typically about **which altitude channel is trusted during replay**, not about toggling between “IGC replay” vs “sensor replay.”

## How major apps and flight computers use the term “replay”

### XCSoar

XCSoar explicitly documents **Logger replay** as replaying **IGC format flight logs** generated by XCSoar or other loggers. During replay it “behaves as if real GPS updates are being received,” i.e., it is fundamentally **flight‑log playback**, not a special “sensor‑only” playback. citeturn4view0

XCSoar also distinguishes replay from a developer-oriented “raw logger” concept (for debugging). While the user manual describes raw logging primarily as an error-tracing tool, community/developer discussions around XCSoar usage show “raw log replay” being used as a way to revisit the whole flight in detail, which implies **a second replay-like workflow** exists in adjacent tooling or builds, and may include device-sourced values (including baro-related fields if present). citeturn4view0turn13search6

### XCTrack

XCTrack’s AIR³ manual documents **IGC replay** as an action originating from “replay this tracklog,” which opens an “IGC replay” menu where you can jump to a flight position and alter replay speed. This is unambiguously **replay of a stored tracklog/IGC**, not replay of raw sensor streams. citeturn12search0turn14search23

XCTrack also documents barometric handling and smoothing for live baro sensor usage (e.g., a low-pass “Smoothing” slider and computed “Lift”), but that is described under sensor calibration/operation, not described as “replay.” citeturn12search1

### LK8000

LK8000 documents that a pilot can “replay my flight” via **Logger Replay**, selecting a logger file. citeturn20view0

More importantly, LK8000’s own manual/change notes contain a highly revealing statement:

> “Fix logger saving pressure altitude and not QNH altitude. The IGC flight replay will use it correctly now.” citeturn16view0

That sentence is a near-direct confirmation of the central idea: **LK8000’s replay consumes the “pressure altitude” field in the IGC** (the barometric/pressure-derived altitude channel), and correctness depends on what the logger wrote.

This is the closest thing in mainstream documentation to what developers and advanced users *mean* when they say “baro-only replay”: replay is still “IGC replay,” but the vertical channel is specifically the **pressure altitude** channel (baro/pressure altitude), not the GPS altitude channel.

## What “baro” means in IGC logs, and why that constrains “baro‑only replay”

The IGC standard (and developer references for it) are explicit: each fix (“B record”) can contain **two different altitude fields**:

- **Pressure altitude** (“Press Alt.” / PPPPP) — altitude to ICAO ISA referenced to **1013.25 hPa** (i.e., pressure altitude / QNE). citeturn6view1  
- **GNSS altitude** (“GNSS Alt.” / GGGGG) — altitude above the **WGS84 ellipsoid**. citeturn6view1

A developer-oriented IGC format reference even walks through an example B-record and states plainly that it contains “altitude from pressure sensor” and “altitude from GPS.” citeturn6view0

### Key consequence

Because IGC stores **pressure altitude**, not raw pressure time series, a “baro-only replay” based on IGC almost always means one of these:

- **Use only the IGC pressure altitude field** when computing displayed altitude and “vario” (vertical speed), or
- **Treat GNSS altitude as unusable or secondary** during replay, e.g., for airspace/flight level checks and vario tone simulation.

It does **not** naturally mean “replay raw barometer sensor readings in Pascals,” because those aren’t in IGC.

### QNH vs pressure altitude is often the hidden gotcha

The LK8000 note above distinguishes **pressure altitude** from **QNH altitude** (altitude corrected to local sea-level pressure). citeturn16view0

That distinction matters for replay and analysis:

- If you replay using **pressure altitude (QNE/1013.25)** you’ll get a consistent “flight level/pressure altitude” track, but not necessarily “true AMSL” without QNH context.
- Some systems/loggers can record events like altimeter setting changes (“ATS”), so software could theoretically reconstruct more “QNH-like” altitude for certain purposes. The IGC reference includes an explicit example of such an event record (“ATS … hPa”). citeturn6view0turn6view1

In practice, many replay engines and analysis tools treat **pressure altitude in IGC** as the authoritative “baro track,” and only apply QNH-like adjustments if they have enough metadata and a consistent policy.

## Where “baro‑only” shows up in real documentation and code-adjacent sources

Even though the exact label “baro-only replay” is uncommon, the underlying concept shows up repeatedly as **baro vs GPS altitude selection**, and as **baro‑only data sources**.

### Replay engines explicitly handle baro-vs-GPS altitude

- **XCSoar release notes** include a fix that explicitly mentions **baro altitude vs GPS during IGC replay**: “baro altitude set by GPS for IGC replay.” This is direct evidence that XCSoar’s IGC replay pipeline tracks a “baro altitude” channel separately, and bugs can arise when those channels get mixed during replay. citeturn21search17turn13search11

- **LK8000** explicitly ties IGC replay correctness to whether the logger saved **pressure altitude** (baro/pressure altitude) vs QNH altitude. citeturn16view0

### “Baro data only” exists as a device/protocol concept

XCTrack’s changelog references “XCTracer support (baro data only…)” and other items like “pure gps altitude into igc,” showing that developers and power users frequently talk about:
- device streams that provide **only barometric data**, and
- configurations where an IGC log might contain **only GPS altitude** (or at least GPS-derived altitude). citeturn21search5turn1search5

This reinforces a practical interpretation:

- “baro-only” is a recognized descriptor for a **data source** (what sensors/fields are available),
- while “replay” is a recognized feature for **playing back logs**, usually IGC.

So “baro‑only replay” is likely **shorthand combining those two axes**: “replay, but the vertical channel is baro-only.”

### Community troubleshooting shows the same model

In LK8000 community support discussions, troubleshooting altitude issues often centers on whether the IGC B-records include both GPS and baro altitude fields, and whether software is configured to prefer one or the other. This is the operational reality behind “baro-only” vs “GPS altitude” replay behavior. citeturn11search7turn16view0

## Replay tools and modes outside the apps

Outside “in-app replay,” there is a large ecosystem of tools that replay, visualize, or stream flight logs:

- Some tools replay IGC by converting it to live NMEA streams (e.g., users discussing replaying an IGC file while outputting NMEA to a COM port to drive glide computer apps for analysis). This is still fundamentally **IGC replay**, not sensor replay—just a different transport layer. citeturn3search19turn14search17

- Online viewers and contest portals often display a **barogram** and sometimes explicitly reference a “baro trace” alongside the map/track. This indicates that many replay/visualization pipelines treat “baro altitude” as a distinct display artifact, usually sourced from the IGC pressure altitude channel. citeturn13search2turn13search5

### Typical replay “modes” you actually see in practice

Across tools, “modes” are usually not labeled “baro-only replay,” but they commonly boil down to:

- **Altitude source selection**: show/use pressure altitude vs GNSS altitude.
- **Speed/time scaling**: 1×, 5×, 20×, skip-to-time.
- **Sensor emulation level**: replay only “position fixes” vs replay “full device sentences” (NMEA) when available.

XCSoar’s manual explicitly warns that very high replay time scales can degrade wind estimation and other computations during replay, which is consistent with “replay as feed of sampled observations into the live algorithms.” citeturn4view0

## Interpreting “on baro‑only replay” in context, and why it matters for app development

### A practical definition for developers and advanced users

When you see a phrase like:

- “works on baro-only replay”
- “fails on baro-only replay”
- “baro-only replay looks clean but full replay is noisy”

the most technically consistent interpretation (given the standards above) is:

> The replay pipeline is feeding the algorithm a vertical channel derived only from pressure/pressure-altitude samples (typically from IGC pressure altitude), without additional vertical observability from GNSS altitude or IMU-derived vertical acceleration, and the behavior differs under that constraint.

This is especially common when:
- testing a **variometer filter** (vario beeps / climb rate),
- validating airspace/flight-level logic where using baro/pressure altitude is preferred, or
- comparing algorithm behavior when **baro is missing**, **baro is zero**, or **GNSS altitude is used as a fallback**.

### What it does *not* usually imply

It usually does **not** mean that raw sensor pressure samples are being replayed, because:
- IGC doesn’t store raw pressure, only pressure altitude. citeturn6view1turn5view0
- “baro-only” is widely used to describe missing GPS/other channels (e.g., certain device protocols), not “replay raw Pa values.” citeturn21search5turn12search1

If you do want raw pressure replay for algorithm tuning (e.g., for pressure-transient QC, Hampel filtering, pressure-rate gating), you generally need **a separate logging format** (raw sensor logs or NMEA sentence logs), not just IGC.

### Implications for variometer app development and flight-data analysis

For a variometer app developer, the term “baro‑only replay” should trigger three immediate technical questions:

1) **What exactly is being replayed?**  
   - IGC fixes (B-record stream) → you have pressure altitude and GNSS altitude, but not raw pressure. citeturn6view1  
   - NMEA/raw logs → you might have device-specific pressure/vario sentences depending on the ecosystem. citeturn13search6  

2) **Which altitude channel is the “truth” during replay?**  
   - Many systems have explicit or implicit preference rules (baro preferred if present; otherwise GPS). XCTrack documents this kind of “prefer baro if available” logic for statistics. citeturn12search0  
   - Replay bugs and fixes show that software often carries both channels and sometimes mistakenly substitutes one for the other during replay. citeturn21search17turn16view0  

3) **What is the replay meant to validate?**  
   - If you’re validating the **vario filter**, you want replay data that matches the filter’s inputs. An IMU+baro EKF can’t be fully validated from IGC alone; “baro-only replay” will only validate the baro leg of the estimator (and any baro QC operating on altitude, not raw pressure). This follows from the IGC field set. citeturn6view1  
   - If you’re validating **airspace flight levels / pressure altitude compliance**, pressure altitude is often the relevant channel; tools and docs explicitly treat pressure altitude as meaningful for those functions. citeturn6view1turn13search6turn13search5  

### A concrete workflow model you can adopt

Below is a practical way to think about the two “replay” families and where “baro‑only” fits.

```text
                         ┌──────────────────────────────┐
                         │          DATA FILE            │
                         └──────────────┬───────────────┘
                                        │
                 ┌──────────────────────┴──────────────────────┐
                 │                                             │
        ┌────────▼────────┐                           ┌────────▼────────┐
        │  IGC REPLAY      │                           │  RAW/NMEA REPLAY │
        │  (tracklog)      │                           │  (device log)    │
        └────────┬────────┘                           └────────┬────────┘
                 │                                             │
     B-record provides:                             provides (varies):
     - lat/lon/time                                 - raw pressure (maybe)
     - pressure altitude (PPPPP)                    - vario estimates
     - GNSS altitude (GGGGG)                        - IMU-derived fields (rare)
                 │                                             │
                 │                                             │
     ┌───────────▼────────────┐                     ┌──────────▼────────────┐
     │ ALTITUDE SOURCE POLICY  │                     │ SENSOR SOURCE POLICY   │
     │ - baro (pressure alt)   │                     │ - baro sensor stream   │
     │ - or GNSS alt fallback  │                     │ - IMU stream, etc.     │
     └───────────┬────────────┘                     └──────────┬────────────┘
                 │                                             │
        ┌────────▼────────┐                           ┌────────▼────────┐
        │ VARIO / NAV      │                           │ EKF / QC / Vario │
        │ computation      │                           │ computation      │
        └──────────────────┘                           └──────────────────┘
```

In this model, “baro‑only replay” most naturally means:
- In the **IGC replay branch**: “use pressure altitude as the only vertical channel.”
- In the **raw/NMEA replay branch**: “feed only baro-related sentences / fields.”

## What I found—and what I did not find

- I found **strong, direct evidence** that replay in major flight software is primarily **IGC/tracklog replay**, and that those systems explicitly care about **pressure altitude vs GNSS altitude** during replay and analysis. citeturn4view0turn12search0turn16view0turn21search17turn6view1  
- I found that “baro data only” is a recognized phrase in device/protocol support contexts (e.g., XCTrack changelog), reinforcing that “baro-only” is about **which data is available**, not necessarily about replaying raw pressure samples. citeturn21search5  
- I did **not** find widely cited official documentation pages using the exact phrase **“baro‑only replay”** as a formal menu item or feature name for XCTrack/XCSoar/LK8000; instead, official sources describe **IGC replay/log replay** plus separate mechanisms describing baro handling. citeturn12search0turn4view0turn20view0  

The evidence therefore supports treating “on baro‑only replay” as **community/developer shorthand** for **flight log replay with a baro-only vertical channel**, usually derived from the **IGC pressure altitude field**, rather than as a standardized term meaning “replay raw barometer sensor data.”
