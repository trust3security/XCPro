# PRD Generation Prompt: Task Save/Load Feature for Gliding App

## Context
You are creating a Product Requirements Document (PRD) for a task save/load feature in a professional gliding app. The app manages flight tasks with waypoints (start, turnpoints, finish) for competitive gliding. Tasks are currently created in the app but need persistent storage management.

## Current System Architecture
- **Task Types**: Racing (FAI rules), AAT (Assigned Area Tasks), DHT (Distance Handicap Tasks)
- **Storage**: Tasks saved as CUP files (standard gliding format) in internal storage
- **UI Location**: Files tab in the task bottom sheet
- **Existing Code**: RacingTaskStorage handles CUP file I/O, TaskManagerCoordinator manages task state

## Feature Requirements

### Core Functionality
1. **Save Current Task**
   - Save active task with user-defined name
   - Overwrite protection (warn if name exists)
   - Auto-suggest name with date/time stamp
   - Quick save with default name option

2. **Load Saved Tasks**
   - Display list of saved tasks with metadata (name, date, waypoints count, distance)
   - Preview task details before loading
   - Replace current task warning
   - Sort by: name, date modified, distance

3. **Delete Tasks**
   - Individual task deletion with confirmation
   - Bulk delete option (select multiple)
   - Show storage space used/freed

4. **Import/Export**
   - Import CUP files from Downloads folder (existing)
   - Export tasks to Downloads for sharing
   - Share task via standard Android share intent

### UI/UX Design (Keep it Simple)
- **List View**: Clean list with task name, date, and key stats
- **Actions**: Swipe to delete, tap to load, long-press for options
- **Save Dialog**: Simple text input with suggested name
- **No Over-Engineering**: Avoid folders, tags, or complex organization

### Technical Constraints
- Use existing CUP format for compatibility
- Store in app's internal files directory (not Downloads)
- Maintain task type separation (Racing/AAT/DHT modules stay isolated)
- File names: alphanumeric with underscores, .cup extension
- Maximum 100 saved tasks (storage management)

### Data to Store
- All waypoints with coordinates and properties
- Task type and configuration
- Cylinder/sector radii and types
- Task name and creation date
- Optional: pilot notes, weather conditions

### User Workflow
1. Pilot creates task by adding waypoints
2. Taps "Save" in Files tab, enters name
3. Task saved to internal storage
4. Later: Opens Files tab, sees saved tasks
5. Taps task to load or swipes to delete
6. Can export task for sharing with others

### Success Metrics
- Tasks persist across app restarts
- Load time < 1 second for typical task
- Zero data loss on save/load cycle
- Compatible with existing gliding software (SeeYou, XCSoar)

## PRD Output Requirements
Generate a concise PRD that:
1. Defines clear user stories
2. Specifies UI components and interactions
3. Lists technical implementation steps
4. Includes error handling scenarios
5. Provides acceptance criteria
6. Stays practical - no unnecessary features

Remember: This is for competitive glider pilots who value reliability and simplicity over feature complexity. The solution should be robust, fast, and intuitive without over-engineering.
